# Project_help-to-Git
## Список базовых команд для работы с **терминалом:**

***pwd ***- показать текущую рабочую папку, из которой выполняется команда; 
***ls*** - отобразить файлы и папки в текущей директории; 
***cd*** - перейти в папку (после cd нужно указать название папки); 
***cd .. *** - перейти на уровень выше; 
***cd ~ *** - перейти в домашнюю папку; 
***mkdir *** - создать папку (после mkdir нужно указать название папки);
***rm *** - удалить файл; 
***rmdir *** - удалить папку; 
***touch *** - создать файл в текущей папке; 
***echo <текст> *** - вывести текст на экран, например, echo "Hello, Practicum!"; 
***echo <текст> >> <файл> *** - вывести текст в файл, например, echo "Hello, Practicum!" >> output.txt; 
***cat <practicum.txt> *** - вывести содержимое файла practicum.txt на экран; 
***mv <источник> <назначение> *** - переместить/переименовать файлы и папки;

##Список базовых команд для работы с **Git**:

***git --version *** - эта команда проверит, установлена ли программа git на ваш компьютер, если да – тогда напечатает версию в следующей строке: например, git version 2.38.1;
***git config --global user.name " <вводите своё имя или ник латиницей и в кавычках> " ***
***git config --global user.email <сдесь нужно ввести свой реальный e-mail> ***
***git config --list *** - вывели в окно командной строки список всех свойств конфига;
***git init *** - Чтобы Git начал отслеживать изменения в проекте, папку с файлами этого проекта нужно сделать Git-репозиторием. Для этого следует переместиться в неё и ввести команду git init (от англ. initialize — «инициализировать»). «Разгитить» папку, если что-то пошло не так, — rm -rf .git cd <папка с репозиторием> # перешли в папку rm -rf .git # удалили подпапку .git
***git status *** - текущее состояние репозитория; 
***git add <и добавляем файл, который нужно записать - например Hello.txt> ***
***git all *** - добавит все файлы к сохранению версии;
***git add .  *** - добавить всю текущую папку;
***git commit -m "<Мой первый коммит>" *** - cделать коммит можно командой git commit с ключом -m(от англ. message – сообщение), который присваивает коммиту название;
***git remote add *** - команда для добавления нового удалённого репозитория; 
***git branch -M main *** - можно читать как «Git, возьми ветку и переименуй её в main». Ключ -M отвечает за переименование, а main — это название самой ветки. 
***git branch *** - показывает все ветки и *-ой показывает на какой сейчас ветке находишься;
***git branch <наззвание ветки>  *** - создает ветку с ее названием, но не переходит сразу на нее;
***git checkout <Название ветки> *** - переключиться на указанную ветку. 
***git checkout <название коммита> *** - мы перейдем к определенному комету! 
***git checkout -b  <Название ветки> *** - создать и переключиться на указанную ветку;
***git merge main  *** - слить ветку (на которой находишься) с веткой main;
***git merge <Название ветки>  *** - мы находимся на ветке main, и Мы сделаем merge ветки <название ветки> в ветку main.
***git rebase main *** - мы находимся на *дополнительной ветке и Теперь изменения из дополнительной ветки находятся в конце ветки main и являют собой линейную последовательность коммитов. Единственная проблема - ветка main не обновлена до последних изменений. Это легко исправить. 
***git rebase <дополнительная ветка> *** - но мы должны перейти на ветку main и уже потом сделать rebase; 
***HEAD *** - это символическое имя текущего выбранного коммита — это, по сути, тот коммит, над которым мы в данный момент работаем. HEAD всегда указывает на последний коммит из вашего локального дерева. Большинство команд Git, изменяющих рабочее дерево, начнут с изменения HEAD.  Отделение (detaching) HEAD означает лишь присвоение его не ветке, а конкретному коммиту. 

***git diff [branch1] [branch2] *** - показать различия между ветками. 

***git log  *** - просмотреть историю коммитов; И посмотреть хеши коммитов! 
###Передвигаться по дереву Git возможно при помощи указания хешей коммитов! Хорошая новость в том, что Git достаточно умён в работе с хешами. Ему нужны лишь первые несколько символов для того, чтобы идентифицировать конкретный коммит. Так что можно написать просто fed2 вместо полного хеша.###

***Git checkout main^ *** - переключимся на коммит Выше main
***Git checkout HEAD^ *** - Можно также использовать HEAD как относительную ссылку. Перемещаемся по дереву коммитов на один комет выше.
***Git checkout HEAD~4 *** -   число коммитов (4), через которые надо пройти.
***git branch -f main HEAD~3 *** - Переместит (принудительно) ветку main на три родителя назад от HEAD. Одна из наиболее распространённых целей, для которых используются относительные ссылки - это перемещение веток. Можно напрямую прикрепить ветку к коммиту при помощи опции -f (Force).

***Git branch -f <main, или любая другая ветка> <Нужный коммит> *** - ЧТОБЫ ПЕРЕМЕЩАТЬСЯ ПО КОММИТАМ ТАК КАК ХОЧЕТСЯ.
Но можно и СНАЧАЛа  ПЕРЕМЕСТИТЬ В НУЖНЫЙ КОММИТ HEAD, а затем уже делать перемещение git branch -f main HEAD

***git stash *** - сохранить изменения в стеке временных изменений; 
### ***git push -u origin main *** ### - указывает Git отправить (или «запушить») изменения из вашего локального репозитория в удалённый. -u – этот ключ означает set upstream: мы устанавливаем связь между локальной веткой и удалённой. В следующий раз, когда вы будете отправлять или получать изменения, Git будет «знать», откуда брать или куда отправлять данные. origin — название стандартного удалённого репозитория. Когда мы сохраняли ссылку на репозиторий командой git remote add, то использовали это короткое название для ссылки на репозиторий. Так Git «поймёт», что ветку нужно искать именно в репозитории с названием origin. main — название ветки, которую вы хотите отправить в удалённый репозиторий.
***git reset HEAD~1 *** - отменяет изменения, перенося ссылку на ветку назад, на более старый коммит, на одну позицию назад. Если нужно на большее количество шагов, то нужно писать HEAD~<число шагов назад> . Это своего рода "переписывание истории"; git reset перенесёт ветку назад, как будто некоторых коммитов вовсе и не было.
***git Revert <название коммита> *** - Чтобы отменить изменения и поделиться отменёнными изменениями с остальными, надо использовать git revert. После revert можно сделать push и поделиться изменениями с остальными.

***git Cherry-pick  - git cherry-pick <Commit1> <Commit2> <...> *** - Это очень простой и прямолинейный способ сказать, что ты хочешь копировать несколько коммитов на место, где сейчас находишься (HEAD).

***git rebase -i HEAD~(число) *** - перенос коммитов туда куда нужно при помощи всплывающего окна, можем промежуточные коммиты либо удалить, либо переставить местами.



##Генерация SSH-ключа
Откройте Terminal и введите команду:
ssh-keygen -t ed25519 -C "<ВАШ EMAIL при регистрации в GitHub>"

Команда выведет в терминал сообщение о готовности сгенерировать приватную и публичную части ключа: > Generating public/private ALGORITHM key pair.
Дальше команда предложит:
1. Ввести путь до папки, где нужно сохранить ключ:
    Enter file in which to save the key (/Users/practicum/.ssh/id_ed25519)
Рекомендуем согласиться с предложенным вариантом — просто нажмите Enter.
2. Ввести парольную фразу для файла:
    Enter passphrase (empty for no passphrase)
Этот пароль будет нужен каждый раз, когда вы попробуете распечатать часть ключа. Программа предлагает создавать ключ и без пароля; чтобы пропустить шаг, нажмите Enter.
3. Повторить парольную фразу ещё раз:
    Enter same passphrase again:
Если вводили на предыдущем шаге пароль — повторите его, если нет – ещё раз нажмите Enter.
Готово! Пара ключей сгенерирована; вы увидите в терминале похожее сообщение:

The key's randomart image is:

|  . o . S  =.++..|
|   o .    o = .o.|
|    .    . . ... |
|              o .|
|            o+ . |

Дальше выполните команду, которая добавит ключ SSH-агенту (программа, которая запоминает ваш приватный SSH-ключ и автоматически подставляет его при подключении к серверу через SSH) и сохранит его в связке ключей (Keychain):
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
Теперь скопируйте публичный ключ id_ed25519.pub в буфер обмена:
pbcopy < ~/.ssh/id_ed25519.pub
        Добавляем SSH-ключ в GitHub аккаунт
1. Зайдите на GitHub и перейдите в Settings (Настройки).
2. В левом меню выберите SSH and GPG keys.
3. Нажмите New SSH key или Add SSH key.
4. В поле Title введите название ключа (например, название вашего компьютера).
5. В поле Key вставьте ключ из буфера обмена горячими клавишами Cmd + V.
6. Нажмите Add SSH key.

##Чтобы вернуться к определённому комиту в Git, нужно выполнить несколько шагов:
Откройте терминал и перейдите в директорию вашего проекта.
Проверьте историю коммитов: выполните команду git log, чтобы увидеть список всех коммитов с их хэшами и сообщениями.
Найдите хэш нужного коммита: это уникальный идентификатор, который выглядит как длинная последовательность букв и цифр.
Переключитесь на нужный комит: используйте команду git checkout <хэш_комита>. После этого вы перейдёте в состояние, соответствующее выбранному комиту.
Создайте новую ветку (опционально): если вы хотите сохранить текущее состояние проекта, создайте новую ветку с помощью команды git checkout -b <имя_ветки>.
После этих шагов ваш проект будет соответствовать состоянию на момент выбранного коммита. Если вам нужно будет вернуться к более поздним изменениям, вы можете переключиться на другую ветку или выполнить дополнительные действия.

            Все основные операции по работе с коммитами, включая возврат к определённому коммиту, выполняются в локальной копии репозитория на вашем компьютере с помощью Git. GitHub же служит для хранения удалённой версии вашего репозитория и совместной работы с другими разработчиками.
Чтобы изменения, сделанные в локальном репозитории (например, возврат к предыдущему коммиту), отразились на GitHub, нужно выполнить следующие шаги:
После того как вы переключитесь на нужный комит в локальной версии репозитория с помощью git checkout <хэш_комита>, вы можете обновить удалённый репозиторий на GitHub. Для этого сначала убедитесь, что вы находитесь в нужной ветке (если вы создали новую ветку, используйте git checkout <имя_ветки>).
Чтобы отправить изменения на GitHub, используйте команду git push origin <имя_ветки> --force. Флаг --force необходим, потому что вы изменяете историю коммитов. Будьте осторожны с его использованием, особенно если над проектом работает несколько человек, так как это может привести к потере данных.
Таким образом, основные действия по работе с историей коммитов всегда выполняются локально, а затем эти изменения можно синхронизировать с удалённым репозиторием на GitHub.
